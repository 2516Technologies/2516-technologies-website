Infrastructure Assessment

What you've built,
what it can do — and
what it can't.

The prototype you and Ari built has done exactly what a prototype should: proven the concept, validated the vision, and given us something concrete to build from. Here's an honest assessment of where it stands today — and what needs to change to support 1,000+ users.

Assessment findings
⚠ Critical — rebuild required
No structured data layer
The prototype appears to store data in flat JSON files rather than a relational database. This is common in rapid prototypes — JSON is fast to work with and requires no database setup. But it means every user's data is essentially a text file, with no ability to query across users, run aggregations, or build any analytics on top of it.
Cannot support concurrent users, analytics, AI training, or any future feature that requires querying across the dataset.
⚠ Critical — rebuild required
CRM has no real backend
The native CRM built into the prototype is a significant piece of UX work — the interface exists and functions in demo. But without a proper database backend, the CRM data isn't persisted reliably, can't scale across thousands of user records, and has no audit trail. It's a frontend without a foundation.
Any data entered into the current CRM cannot be safely migrated or relied upon at scale. A backend rebuild is not optional.
⚡ Risk — needs architecture decision
Node.js runtime — unconfirmed structure
The prototype is likely running on Node.js, which is a perfectly capable runtime for this type of application. However, without reviewing the actual codebase, the architecture is unclear — it may be a single monolithic file, an Express app, or something built with a no-code tool that generates Node. The structure matters for what can be reused vs. rebuilt.
Upload the prototype file so 2516 can do a full code review before the proposal is finalized. This directly affects Phase 1 timeline and cost.
⚡ Risk — needs resolution
No mobile-first design
Based on the description of the prototype, it was built to demonstrate the concept rather than optimize for mobile. For a platform targeting small nonprofit staff — who frequently work from phones — a desktop-first or desktop-only UI is a significant barrier to adoption. Mobile is not a nice-to-have; it's the primary use case.
Phase 1 will rebuild the UI mobile-first from the ground up, preserving the design intent while making every surface work on a 375px screen.
⚡ Risk — plan required
No authentication or multi-tenancy
A prototype typically has minimal or no authentication — often a hardcoded password or no login at all. For a subscription platform serving 1,000+ organizations, multi-tenant architecture is essential: each organization's data must be completely isolated from every other organization's data. This requires a purpose-built auth and tenancy system.
2516 will implement AWS Cognito for authentication and row-level security in Aurora for tenant isolation. No user data can ever cross organizational boundaries.
✓ Strong — preserve and build from
Claude API integration works
The core AI interaction — using Claude's API to power the virtual consultant and chatbot experience — has been validated in the prototype. The prompt architecture and conversation patterns you and Ari developed are real, tested, and carry value. This is the hardest part to validate, and you've already done it.
The Claude API integration and prompt engineering from the prototype will inform Phase 2's RAG chatbot design. The concept is proven — now we give it infrastructure.
✓ Strong — preserve and build from
UX vision is clear and differentiated
The prototype demonstrates a clear vision for what the product should feel like — the CRM interface, the virtual consultant interaction model, and the overall flow reflect genuine product thinking. That vision is the most valuable output of the prototype phase and it translates directly into the design brief for Phase 1.
2516 will present three visual design directions based on the prototype's UX intent before any code is written. You approve the direction before development begins.
✓ Opportunity — use it wisely
You have a working demo
Most platforms at this stage have nothing to show. You have a functioning prototype that communicates the vision to stakeholders, investors, and early clients. That's a real asset for the RFP process, early sales conversations, and board presentations — even before the production platform is built.
Use the prototype for demos while Phase 1 is built. Never deploy it to real users — but use it to generate revenue conversations and validate subscription interest before launch.

The scaling wall — why this matters for 1,000 users

Flat JSON storage works fine for one or two users testing a concept. It begins to fail somewhere between 10 and 50 users, and it collapses entirely at scale. Here's specifically what happens as you grow:

Feature / RequirementCurrent stateWhat 1,000 users needs
User data storageFlat JSON files — one per user or sessionRelational database with indexed tables, foreign keys, and query support
Concurrent usersFile read/write conflicts at any meaningful concurrencyConnection-pooled database that handles hundreds of simultaneous queries
Analytics & reportingImpossible — no way to query across JSON filesSQL queries across normalized tables; behavior tracking; dashboards
AI model training (Year 2)Nothing to train on — unstructured, non-queryable data12+ months of clean, labeled, structured interaction data
Subscription billingNo billing infrastructure existsStripe integration tied to user records in the database
Data security / tenant isolationNo separation between user dataRow-level security ensuring Org A can never see Org B's data
Audit trailNo logging or activity historyTimestamped event log for every user action — required for compliance

What we're keeping — and why

This is not a "throw everything out" situation. The prototype did its job: it validated the concept, proved the Claude API integration works, and produced a UX vision that drives the Phase 1 design brief. None of that work is wasted. The rebuild is about giving a proven idea the infrastructure it deserves — not starting from scratch.

Think of it this way: the prototype is the architectural sketch. Phase 1 is the structural engineering. The vision stays the same. The foundation changes underneath it so that everything you want to build in Years 2 and 3 is actually possible.

The path forward
1

Code review of the prototype

Once the prototype file is shared, 2516 conducts a full technical review — confirming the runtime, mapping what can be reused (prompt logic, UX patterns), and documenting exactly what needs rebuilding. This informs the final Phase 1 scope and timeline before any contract is signed.

Phase 1 builds the foundation

Proper data architecture, mobile-first UI, CRM rebuilt on Aurora, email/text notifications, auth, and subscription billing. Every milestone is demoed and approved before the next phase begins. Nothing ships to real users until it's ready.

4

AI features in Phase 2 — after the foundation is solid

The virtual consultant, interactive avatar, and Claude-powered chatbot are Phase 2. They require Phase 1's data layer, user session infrastructure, and stable platform to function correctly. Skipping ahead risks delivering exactly what CJ is most concerned about: something that doesn't work.
